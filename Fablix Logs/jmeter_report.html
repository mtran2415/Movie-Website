Case 1: HTTP/1 thread 98 6.99 6.85 Each request is processed relatively fast, since we are only processing a single request at a time, however as a result the throughput is rather low.
Case 2: HTTP/10 threads 120 27.4 27.1 Slower than using 1 thread since we aren't balancing the load at all; the server must process every request given to it without distributing them elsewhere. Also, slower than Case 1 since there are more user connections.
Case 3: HTTPS/10 threads 126 26.3 26.2 Similar to Case 2, however using HTTPS does introduce some additional overhead.
Case 4: HTTP/10 threads/No prepared statements 122 29.4 29.2 Slower than using prepared statements, since the same statement is getting validated more than once (same statement, different parameters, many times).
Case 5: HTTP/10 threads/No connection pooling 129 36.1 35.9 Slowest of all the cases since the connections are not getting cached: a brand new connection needs to be made for each request.

Case 4: HTTP/10 threads/No connection pooling 108 15.5 15.2 As was the case with the Single Instance, having no connection pooling significantly increases the amount of time needed for each query, since connections are not cached.
